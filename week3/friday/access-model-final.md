Access Model Final
Overview

The KijaniKiosk production foundation uses a least-privilege access model. Each service runs under its own dedicated service account and can access only the files required for its function. Shared resources are controlled through group membership and ACLs rather than broad filesystem permissions.

Service Account Design
kk-api

Purpose:

Runs the API service.
Reads required configuration.
Writes application logs.

Restrictions:

Cannot access payments application files.
Cannot modify system configuration.
Cannot obtain root privileges.

Reasoning:
A dedicated account limits the impact of compromise of the API service.

Configuration Access

Configuration files are stored under:

/opt/kijanikiosk/config

Ownership:

root:kijanikiosk
750

Reasoning:

Basic UNIX permissions protect configuration files from modification by service accounts while allowing controlled read access through group membership and ACLs.

ACLs were chosen because multiple services require read access to different configuration files. Traditional owner/group permissions alone would not provide sufficient granularity.

Shared Log Access

Directory:

/opt/kijanikiosk/shared/logs

Ownership:

kk-logs:kk-logs
2770

ACLs:

kk-api: rwx
kk-payments: r-x
louiemimo: r-x

Reasoning:

The API service must create and update logs.

The payments service requires read-only access for audit correlation and troubleshooting.

The administrator requires read access for operational support.

The setgid bit ensures newly created files inherit the logging group automatically.

ACL Design Rationale

ACLs were selected for shared logging because standard UNIX permissions cannot provide:

Write access for kk-api.
Read-only access for kk-payments.
Administrative read access.
Central ownership by kk-logs.

ACLs provide precise permission assignments without weakening directory security.

Health Directory Design

Directory:

/opt/kijanikiosk/health

Ownership:

kk-logs:kijanikiosk
750

Health file:

last-provision.json

Ownership:

kk-logs:kijanikiosk
640

Reasoning:

The provisioning process writes health-check results that must be available to monitoring and operational users.

The health directory was added during the Friday production foundation project because it was not part of the original Tuesday access model.

The chosen ownership allows monitoring data to be generated by the logging service while remaining readable by authorized group members.

Logrotate Compatibility

A critical requirement is that the access model survives log rotation.

The shared log directory uses default ACLs and the setgid bit so that newly created log files inherit the correct ownership and permissions after rotation.

Verification was performed by:

Forcing a log rotation.
Creating a new file as kk-api.
Confirming write access remained functional.

Result:

PASS — kk-api retained write access after rotation without manual permission repair.

Security Benefits
This access model enforces least privilege, prevents cross-service access, supports operational monitoring, preserves audit visibility, and remains functional after maintenance tasks such as log rotation and service restarts.
